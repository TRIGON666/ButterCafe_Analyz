# Пошаговая инструкция: локальная аналитическая система для ButterCafe

## 1. Что делаем и какой будет результат

Вы превращаете текущий сайт-витрину в локальную аналитическую систему, которая работает на одном компьютере (или в локальной сети) без Yandex Cloud и других облаков.

Финальный результат:

- Сайт ButterCafe работает локально на PostgreSQL.
- Есть регистрация и авторизация клиентов.
- Есть личный кабинет с историей заказов и профилем.
- Собираются данные для аналитики.
- Ежедневно формируется локальный отчет для владельца по email.
- Есть дашборды в Metabase (локально).

## 2. Текущее состояние проекта

Уже есть:

- Django-проект и приложение кафе.
- PostgreSQL в конфигурации.

## 3. Локальный стек (замена облачным сервисам)

Используем локальные аналоги:

- App Platform -> Django runserver или локальный Gunicorn/Nginx.
- Managed PostgreSQL -> локальный PostgreSQL.
- Object Storage -> локальная папка data_lake или локальный MinIO.
- Cloud Functions -> локальный планировщик (Windows Task Scheduler или APScheduler/Celery Beat).
- Metabase в облаке -> Metabase в Docker локально.

## 5. Этап 1. Личный кабинет и клиенты

### 5.1 Подключить регистрацию и авторизацию (django-allauth)

1. Установить пакет:

        pip install django-allauth

2. Добавить пакет в [buttercafe/settings.py](buttercafe/settings.py):

- В INSTALLED_APPS добавить allauth-приложения.
- В MIDDLEWARE добавить AccountMiddleware.
- Указать SITE_ID = 1.
- Добавить AUTHENTICATION_BACKENDS для allauth.
- Настроить ACCOUNT_LOGIN_METHODS, ACCOUNT_SIGNUP_FIELDS и редиректы LOGIN_REDIRECT_URL, LOGOUT_REDIRECT_URL.

3. Подключить маршруты allauth в [buttercafe/urls.py](buttercafe/urls.py).
4. Выполнить миграции и проверить регистрацию/вход/выход.

### 5.2 Привязать заказ к пользователю

Сейчас заказ создается без авторизованного пользователя. Нужно:

1. В модель Order добавить поле user (ForeignKey на auth.User, null=True, blank=True).
2. В логике оформления заказа (создание Order) подставлять request.user, если пользователь авторизован.
3. Сделать миграцию и проверить оформление заказа в двух режимах:

- Гость
- Авторизованный пользователь

### 5.3 Страницы ЛК

Рекомендуемая структура:

- Профиль: имя, телефон, email, адрес доставки по умолчанию.
- История заказов: список заказов пользователя.
- Повтор заказа: кнопка Перезаказать, которая копирует позиции в корзину.
- Избранное (опционально): отдельная модель favorite user<->product.

Что сделать по коду:

1. Добавить маршруты и view-функции для ЛК в [cafe/urls.py](cafe/urls.py) и [cafe/views.py](cafe/views.py).
2. Добавить шаблоны страниц ЛК в каталог templates приложения.
3. Ограничить доступ к ЛК через декоратор login_required.

Проверка этапа:

- Новый пользователь регистрируется и входит.
- В ЛК видит только свои заказы.
- Кнопка повтора заказа снова добавляет товары в корзину.

## 6. Этап 2. Сбор данных для аналитики

### 6.1 Обязательные изменения моделей

По вашему ТЗ:

- Добавить Order.created_at с временем.
- Добавить Product.cost_price.

Практически лучше сделать так:

1. Переименовать существующее поле Order.created -> Order.created_at через миграцию RenameField.
2. Обновить сортировку и места использования поля в коде.
3. Добавить в Product поле cost_price (DecimalField, null=True на период заполнения).
4. Ввести в админке контроль: cost_price <= price.

### 6.2 События регистраций и действия пользователей

Минимальный состав данных:

- Регистрация пользователя (кто, когда).
- Вход в систему.
- Добавление в корзину.
- Оформление заказа.

Реализация:

1. Создать модель EventLog с полями timestamp, user, event_type, metadata_json.
2. В критичных view-функциях писать события в EventLog.
3. Для регистрации/логина подключить Django signals.

### 6.3 Ежедневная выгрузка в локальное хранилище

Вместо Object Storage из облака используйте один из вариантов:

- Простой: папка data_lake внутри проекта.
- Ближе к прод-архитектуре: локальный MinIO.

Что выгружать ежедневно:

- Заказы (JSON и CSV).
- События регистраций.
- Логи действий пользователей.

Как сделать:

1. Создать management command export_daily_analytics.
2. Экспортировать данные за предыдущий день в папку вида:

        data_lake/YYYY/MM/DD/orders.csv
        data_lake/YYYY/MM/DD/orders.json
        data_lake/YYYY/MM/DD/events.csv

3. Запускать по расписанию в 08:00 через Windows Task Scheduler:

        schtasks /Create /SC DAILY /TN "ButterCafeDailyExport" /TR "cmd /c cd /d C:\path\to\ButterCafe-main && .venv\Scripts\python manage.py export_daily_analytics" /ST 08:00

## 7. Этап 3. Дашборды и отчетность (локально)

### 7.1 Поднять Metabase локально

Способ через Docker:

    docker run -d --name metabase -p 3000:3000 metabase/metabase

После запуска открыть:

    http://localhost:3000

### 7.2 Подключить источники данных

Минимум:

- Основной источник: PostgreSQL (операционные таблицы).

Опционально:

- Архивный слой: данные из data_lake, предварительно загружаемые в отдельные таблицы analytics_.*.

### 7.3 Дашборды, которые нужны владельцу

Создать 4 дашборда:

1. Продажи
- Выручка по дням.
- Количество заказов.
- Средний чек.

2. Товары
- Топ товаров по выручке.
- Топ товаров по количеству.
- Маржинальность: price - cost_price.

3. Клиенты
- Новые клиенты по дням.
- Повторные покупки.
- RFM-анализ (Recency, Frequency, Monetary).

4. Время
- Часы пик по заказам.
- Дни недели с максимальной выручкой.

## 8. Этап 4. Автоматизация ежедневного отчета

Вместо Yandex Cloud Functions сделать локальный сервис отчета:

1. Команда generate_daily_report:

- Запрашивает метрики из Metabase API.
- Формирует текст отчета за вчера.
- Отправляет email владельцу.

2. Настроить SMTP в .env:

- EMAIL_HOST
- EMAIL_PORT
- EMAIL_HOST_USER
- EMAIL_HOST_PASSWORD
- EMAIL_USE_TLS
- OWNER_REPORT_EMAIL

3. Запуск по расписанию в 08:00 отдельной задачей Windows:

        schtasks /Create /SC DAILY /TN "ButterCafeDailyReport" /TR "cmd /c cd /d C:\path\to\ButterCafe-main && .venv\Scripts\python manage.py generate_daily_report" /ST 08:00

Пример формата письма:

- Выручка вчера: 25 430 руб (+15%).
- Топ-3 товара: Круассан (12), Тирамису (8), Эклер (7).
- Новых клиентов: 5.

## 9. Этап 5. Контроль качества и приемка

Перед сдачей сделать чек:

1. Функциональные проверки
- Регистрация/вход/выход работают.
- ЛК доступен авторизованному пользователю.
- История заказов и повтор заказа работают.

2. Проверки данных
- Поля created_at и cost_price заполнены и используются в отчетах.
- Ежедневный экспорт файлов создается без ошибок.
- Логи событий фиксируются.

3. Проверки аналитики
- Metabase подключен к БД.
- Все 4 дашборда обновляются актуальными данными.
- Утренний отчет приходит на email.

## 10. Локальные критерии готовности (DoD)

- Сайт работает локально на PostgreSQL.
- ЛК с историей заказов реализован.
- Дашборды показывают актуальные показатели.
- Ежедневные отчеты автоматически приходят на почту.
- Все компоненты связаны в единую локальную систему.

## 11. Приоритетный порядок разработки (рекомендуемый)

1. Авторизация и ЛК.
2. Привязка заказа к пользователю.
3. Поля created_at и cost_price.
4. EventLog и экспорт в data_lake.
5. Metabase и дашборды.
6. Автоотчет по email в 08:00.

Такой порядок снижает риск: сначала собираете корректные данные, затем строите дашборды и только потом автоматизируете рассылку.
